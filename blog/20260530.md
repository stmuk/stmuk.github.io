The purpose of this post is to explain how to run text-based AI models on your
own computer without relying on someone else’s server. The software I use is
**llama.cpp**. While there are other pieces of software available, they are
generally just front-ends that contain a copy of llama.cpp inside them;
therefore, I recommend using llama.cpp directly.

Because the local LLM space changes rapidly, I suggest checking out the
software regularly to get the very latest version. New features are added on a
weekly basis, which can significantly improve the speed of your model. Another
piece of software is called **Pi** (not the Raspberry Pi). It is the Pi coding
agent, which queries your local LLM to enable guided tasks like coding and web
searching. To keep up with these rapid changes, I suggest monitoring social
media, specifically **X** (formerly Twitter where most developers post), and
**Hugging Face**, the main site where you download model data.

These models won't be as good as ChatGPT, Claude and Gemini but they are good
enough to replace them for many tasks and the rate of improvement is high.

## Selecting and Testing Models

llama.cpp essentially acts as a query tool for a large database or model.
Models are released daily so you should also check blogs regularly.  The main
blog to follow is [Simon Willison’s](https://simonwillison.net/).
Also [Reddit's LocalLLaMA](https://old.reddit.com/r/LocalLLaMA) is worth
checking as well as Hugging Face itself for the release of new models. There is
usually considerable hype around new models, with everyone claiming they are
the best thing ever. While this hype is sometimes justified, it
is more frequently not. I suggest sticking with models that have been released
for a period of time and have received updates. 

Currently, the leading supplier of open-source LLMs tends to be Chinese, with
**Qwen** probably being the leader.  There are some downsides to using Chinese
models. They operate under Chinese law and are often politically censored if
you ask questions about topics like the Tiananmen Square protests, they will
often refuse to answer or even be rude. Additionally, they can be heavily
biased toward Marxist theory, which I regard as unnecessary. Other good models
generally come from Google’s **Gemma4** series.

People frequently uncensor these models so the model answers more honestly
without refusing topics. However, these uncensored models often don’t behave as
well as the original since the censorship removal process can cause side
effects akin to "brain damage". I keep a bunch of different models around for
different purposes, including specific coding models, which are useful for
programming tasks where censorship doesn’t matter.  [Unsloth modified
models](https://huggingface.co/unsloth) are particularly good with examples
below.

## Managing Files and Hardware

The file names themselves tend to be confusing, consisting of long strings of
acronyms. I have provided a table below to help you interpret them. Generally,
you will need a file ending in **.gguf** to use with llama.cpp. This is a
single-file format. While other formats exist that use multiple files, I find
the single-file GGUF format much easier to manage. I also suggest renaming
the downloaded file to include the repository name from Hugging Face, as the
base file name often makes it difficult to trace its origin. Furthermore, take
notes on which models you find best, as you will end up downloading so many
that you will lose track.  If you login your download speed will be faster.

I recommend creating a small personal selection of obscure questions to test and
benchmark the LLMs. Try to think of questions that are unlikely to have been
asked by others, as public test cases may eventually be included in training
data. For example, asking about "Simon Willison and pelican riding a unicycle"
is likely too optimized by LLM suppliers because he is well-known. Keep your
testing data quiet.

The main ways to to run LLMs on hardware are: GPU, ideally an external graphics
card with plenty of VRAM and CPU, which is slower and uses main RAM not VRAM
but better than nothing, The two main suppliers are **Nvidia** (superior
support) and **AMD** (cheaper but tends to be more problematic).

My personal use has been low-end AMD GPUs integrated into the CPU (iGPU).  The
AMD brand is Radeon, Intel Iris Xe and also Apple Silicon.  These use main
RAM rather than VRAM.

There are essentially two sets of AMD drivers: the official ROCm driver
(notorious for being a world of pain) and an alternative called Vulkan. Vulkan
seems to have better support for a wide range of hardware than ROCm and still
offers a significant speedup.  I use a simple shell script with `cmake`
commands to configure it to use Vulkan. You will also need to install the
correct drivers for your OS.  Intel should also work with Vulkan.  Apple
Silicon has particularly good support in llama.cpp and needs no special compile
flags.

There are quite a large number of compile-time and runtime flags for the
software, which can be confusing. I keep these in shell scripts to track them,
as these settings change on a weekly basis. After compiling, llama.cpp contains
several different binaries. The main one is `llama-server`, which runs a web
server on port 8000 that resembles the OpenAPI API standard. There is also a
command-line chat interface, but they take slightly different flags.  I suggest
focusing on the server version because it is easier to integrate with other
software.

Finally, you’ll need to match things inside the file name with the runtime
flags. For example, if the file has "MTP" or specific quantization tags, you
may need specific flags to achieve the expected speed increase. Models come in
at least two forms: dense models and Mixture of Experts (MoE). A basic metric
associated with any model is the number of parameters (e.g., a 7B model has 7
billion parameters). This is roughly related to the amount of RAM you have
available. For instance, a 7B model should roughly fit into 32 GB of RAM,
though there are many online calculators to help you determine the exact
requirements.

Most prefer Q4_K_M (4-bit quantization) as it offers an optimal balance
of speed, quality, and memory efficiency. At this precision, 7–9B parameter
models easily fit into 16GB of RAM, while 27B parameter models require
approximately 32GB of RAM. Note that Mixture of Experts (MoE) models are
different because they possess two sets of parameters (total and active), which
affects how their size is calculated.

## Install and Config

### Vulkan llama.cpp 

- Dependencies under Debian-type systems
```
sudo apt install build-essential cmake ccache vulkan-tools libvulkan1 libvulkan-dev glslc spirv-headers
```

- Compile
```
cmake -B build -DCMAKE_BUILD_TYPE=Release -DGGML_VULKAN=ON
cmake --build build -j$(nproc)
```

### CPU llama.cpp
```
cmake -B build -DCMAKE_BUILD_TYPE=Release -DGGML_NATIVE=ON -DGGML_BLAS=ON -DGGML_BLAS_VENDOR=OpenBLAS
cmake --build build -j$(nproc)
```

### Apple Silicon llama.cpp (or alternative CPU for benchmarking)
```
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j$(nproc)
```

- Add binaries under build/bin to PATH

### Useful flags

Check upstream docs since these frequently change!

Turn thinking off (not always respected by models)
```
--reasoning off
```

Turn on MTP for speed boost (needs MTP model)
```
--spec-type draft-mtp --spec-draft-n-max 3 \
```

Context length (useful to increase for agent use)

Models have a default context visible with -v

```
-c 131072 
```

### llama-server

- Binds to localhost 8080 with an OpenAPI compatible API.

examples:

```
# MTP like (doesn't work with non MTP models)
llama-server -v -m $1 \
  --host 0.0.0.0 \
  --gpu-layers 99 \
  --reasoning off \
  --flash-attn on \
  --batch-size 2048 \
  --ubatch-size 512 \
  --threads 16 \
  --cache-type-k q4_0  \
  --cache-type-v q4_0 \
  --spec-type draft-mtp --spec-draft-n-max 3 \
  --spec-default  
```

```
# GPU like
llama-server \
    -m $1 \
    --gpu-layers 99 \
    -c 131072 \
    --reasoning off \
    --flash-attn on \
    --cache-type-k q4_0 \
    --cache-type-v q4_0 \
    --spec-default
```

### llama-chat

Command line chat client.

examples:

```
# CPU like
llama-cli \
    -m $1 \
    --no-mmap \
    --threads 6 \
    --reasoning off \
    --flash-attn on \
    --cache-type-k q4_0 \
    --cache-type-v q4_0 \
    --spec-default
```

### Benchmarking

It is recommended to run benchmarks to determine the optimal parameters. There
is a tool called `llama-bench` that is particularly useful for CPU testing.
The number of CPU threads is an important parameter. Generally, you should set
this to match the number of physical or performance cores, but you can run a
few benchmarks to find the best setting for your specific hardware.

```
llama-bench -m Qwen2.5-Coder-7B-Instruct-Q4_K_M.gguf --n-prompt 1 --n-gen 1 -t "$THREADS" --mmap 0 
```

This example uses impatient flags and doesn't time for very long!  It returns
two metrics one for the prompt processing (usually less important) and the
"tg1" is usually more useful.

### Filename Tags Table

| Filename component | Example | Meaning |
|-------------------|---------|---------|
| `26B` | `Gemma-4-26B` | 26 billion parameters |
| `A4B` | `26B-A4B` | MoE active parameters. Example: 26B total parameters, ~4B active per token. |
| `abliterated` | `Qwen3-30B-Instruct-abliterated` | Uncensored. |
| `Coder` | `Qwen3-Coder-30B` | Optimized for programming tasks trained on code data. |
| `heretic` | `Gemma-4-it-heretic` | Uncensored by Heretic software. |
| `Instruct` / `it` | `Gemma-4-31B-it` | Instruction-tuned model. Trained to follow prompts and chat. |
| `IQ4` | `IQ4_XS`, `IQ4_NL` | Importance-aware 4-bit quantization. Often preferred over standard Q4 at similar sizes. |
| `mmproj` | `mmproj.gguf` | Multimodal projector file. Extra file required by vision models for image understanding.. |
| `MoE` | `Mixtral-8x7B`, `26B-A4B` | Mixture-of-Experts architecture. Only part of the network is active for each token, reducing compute cost. |
| `MTP` | `mtp-model.gguf` | Multi-Token Prediction model. Used for speculative decoding or draft-token generation to increase inference speed. |

## pi coder

[https://pi.dev/](https://pi.dev/)

* pi coder is a terminal app which talks to the local model endpoint.

* It can read and write files and search the web.

* It's usually used for programming but can do job searches etc.

* Openclaw is based on pi but pi is more lightweight and low level.

* pi should be secured by bubblewrap on Linux and macOS sandboxing to only access particular files 

local config
copy to `.pi/agent/models.json`
```
{
  "providers": {
    "llama-cpp": {
      "baseUrl": "http://localhost:8080/v1",
      "api": "openai-completions",
      "apiKey": "none",
      "models": [
        { "id": "your-model-name" }
      ]
    }
  }
}
```

* Sadly not all models work well with pi

### Uses of Small LLMs

* Privacy since no data leaves your machine.

* Saving money on large model tokens.

* Checking and rewriting text (helped with the production of this article and fixing my broken markdown).

* Basic text processing as an alternative to Perl and awk.

* Basic programming code review.

* Generating boilerplate on new programming projects.
